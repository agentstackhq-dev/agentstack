# Environments

This template defines `development`, `preview`, and `production`.

Declare required custom environment values in `agentstack.config.json` under `env.custom`. Each declaration must include explicit `providerTargets` entries that name the owning provider service, surfaces, environments, and source for provider sync.
For this prototype, local validation reads actual values from `.agentstack/env-values.json` when the file exists. Treat that file as CLI-managed local state instead of editing it by hand.

Use `agentstack env set` to write values into local validation state only. The CLI validates the environment, surface, declared variable name, declaration scope, and enum values before updating `.agentstack/env-values.json`:

```bash
agentstack env set --env preview --surface convex --name STRIPE_MODE --value sandbox
```

Use `agentstack env inspect --env preview` or `pnpm run env:inspect` to confirm whether declared bindings are present without exposing secret values.

Use `agentstack inspect --env preview` or `pnpm run inspect` to see provider adapter contract status and pending provider operation IDs. `contract-only` means the provider boundary is normalized, but mutations are still local-cloud rehearsal here. `clerk:command-plan`, `convex:command-plan`, `vercel:command-plan`, and `eas:command-plan` mean Agentstack can plan current provider CLI command shapes. Runtime provider execution is available only through explicit `agentstack provider inspect/apply` commands. Operation IDs are stable and redacted; env operations include surface scope and variable names only, never values or hashes. An `env.set` operation can appear before a local value is available; sync remains the actionability gate.

Missing `.agentstack/env-values.json` is treated as an empty value set. Invalid JSON or non-string values fail `validate`, `validate --quality`, and `validate --cloud`.

Use `agentstack sync --env <env>` after local values exist to reconcile the local provider env resource rehearsal. Sync refuses missing or invalid declared values before planning or applying provider env resources. The local-cloud adapter creates provider env resources only from declared `providerTargets`, such as `convex.convex.STRIPE_MODE` when `STRIPE_MODE` targets the Convex provider service on the Convex surface. Applied sync writes those resources to `.agentstack/local-cloud.json` with redacted metadata and `valueHash` only; raw env values are never written there.

Use `pnpm run provider:preview:plan` or `pnpm run provider:production:plan` to print one aggregate command plan for every enabled provider in manifest order. It runs local validation first, stops before service sections if validation fails, and prints `PLAN provider <env> all`, `Evidence: provider-command-plan`, `Provider execution: none`, `Mutation: none`, and `Readiness: not-claimed`. The aggregate plan does not read live provider state, does not call provider CLIs, does not write `.agentstack/provider-links.json`, `.agentstack/local-cloud.json`, telemetry, provider resources, or `docs/provider-resource-ledger.md`, and refuses ambiguous `--all --service ...` input.

Use `pnpm run provider:preview:reconcile` or `pnpm run provider:production:reconcile` to print one aggregate reconciliation plan artifact for every enabled provider in manifest order. It runs the same local validation gate first, stops before service sections if validation fails, and prints `PLAN provider reconcile <env>`, `Evidence: provider-reconciliation-plan`, `Provider execution: none`, `Mutation: none`, `Readiness: not-claimed`, `Current source: local-validation-and-ledger-only`, `Live state: not-read`, and `Local-cloud state: not-read`. This artifact is plan-only: it does not read live provider state, local-cloud rehearsal state, or provider links; does not call the provider executor or provider CLIs; does not write `.agentstack/events.jsonl`, `.agentstack/provider-links.json`, `.agentstack/local-cloud.json`, telemetry, provider resources, or `docs/provider-resource-ledger.md`; and does not claim readiness, identity, drift, provisioning, adoption, links, or live coherence. Per-service rows summarize desired enabled state, unknown current state, ambiguous identity, unproven drift, sanitized ledger status, `Operations: not-evaluated`, command count, and a next evidence-gathering command for the selected environment.

Use `pnpm run provider:clerk:preview` to print a deterministic Clerk preview command plan with `Evidence: provider-command-plan` without running provider commands. Generated projects include the Clerk package so `pnpm exec clerk` resolves locally. Preview planning prints `pnpm exec clerk init -y`, `pnpm exec clerk doctor --mode agent`, `pnpm exec clerk env pull --mode agent`, and `pnpm exec clerk config pull --mode agent`.

Use `pnpm run provider:clerk:inspect:preview` or `agentstack provider inspect --service clerk --env preview` for explicit read-only Clerk diagnostics with `Evidence: live-read`. Clerk inspect requires an authenticated Clerk CLI or `CLERK_SECRET_KEY`, depending on local Clerk setup. Keep raw key values out of command examples, logs, docs, and telemetry. Development environment inspect is rejected. Clerk apply is unavailable in this slice.

Use `pnpm run provider:clerk:production` to print a deterministic Clerk production command plan without running provider commands. Production planning requires `CLERK_SECRET_KEY`, prints the same inspection commands plus `pnpm exec clerk deploy --mode agent`, and marks production confirmation as required for the command-plan surface.

Use `pnpm run provider:convex:preview` to print a deterministic Convex preview command plan with `Evidence: provider-command-plan` without running provider commands. Generated projects include the Convex package so `pnpm exec convex` resolves locally. Preview planning requires `CONVEX_DEPLOY_KEY` and prints `pnpm exec convex deploy --preview-name __APP_SLUG__-preview`. Preview env commands use `.agentstack/env-values.json` as the value source label and `convex env --deployment <preview-deployment-name> set/remove ...` until a concrete Convex preview deployment exists.

Use `pnpm run provider:convex:inspect:preview`, `pnpm run provider:convex:inspect:production`, or `agentstack provider inspect --service convex --env <preview|production>` for explicit Convex read/diagnostic provider interaction with `Evidence: live-read`. This runs only `pnpm exec convex env --deployment <preview-deployment-name> list` for preview or `pnpm exec convex env --prod list` for production.
For explicit preview execution, use `pnpm run provider:convex:apply:preview` or `agentstack provider apply --service convex --env preview`; supported apply prints `Evidence: live-mutation`.
Convex inspect and apply require `CONVEX_DEPLOY_KEY`. Development environment inspect/apply is rejected. Production Convex inspect is read-only; production Convex apply requires `--confirm-production`.

Use `pnpm run provider:convex:production` to print a deterministic Convex production command plan without running provider commands. Production planning requires `CONVEX_DEPLOY_KEY`, prints `pnpm exec convex deploy`, scopes env commands with production Convex flags, and marks production confirmation as required for the command-plan surface.

Use `pnpm run provider:vercel:preview` to print a deterministic Vercel preview command plan with `Evidence: provider-command-plan` without running provider commands. Generated projects include the Vercel package so `pnpm exec vercel` resolves locally. Preview planning requires `VERCEL_TOKEN`, a linked Vercel project from `vercel link` or `.vercel/project.json`, and prints `pnpm exec vercel deploy --target=preview`.

Use `pnpm run provider:vercel:inspect:preview`, `pnpm run provider:vercel:inspect:production`, or `agentstack provider inspect --service vercel --env <preview|production>` for explicit read-only Vercel provider inspection with `Evidence: live-read`. This runs only `pnpm exec vercel env ls <env>` and `pnpm exec vercel project ls --json` through the provider executor. Vercel deploy execution, env add/update/remove execution, and production apply remain unavailable in this inspect path.

Use `pnpm run provider:vercel:apply:preview` or `agentstack provider apply --service vercel --env preview` for explicit Vercel preview provider execution with `Evidence: live-mutation`. This runs only `pnpm exec vercel deploy --target=preview` through the provider executor. Vercel env add/update/remove commands remain plan-only, and Vercel production apply is unavailable in this slice.

Use `pnpm run provider:vercel:production` to print a deterministic Vercel production command plan without running provider commands. Production planning requires `VERCEL_TOKEN`, requires the same linked Vercel project, prints `pnpm exec vercel --prod`, and marks production confirmation as required for the command-plan surface.

Use `pnpm run provider:eas:preview` to print a deterministic EAS preview command plan with `Evidence: provider-command-plan` without running provider commands. Generated projects include `eas-cli` so `pnpm exec eas` resolves locally. Preview planning requires `EXPO_TOKEN`, prints `pnpm exec eas project:init --non-interactive`, `pnpm exec eas env:list --environment preview`, and `pnpm exec eas build -p all -e preview --json --non-interactive`.

Use `pnpm run provider:eas:inspect:preview`, `pnpm run provider:eas:inspect:production`, or `agentstack provider inspect --service eas --env <preview|production>` for explicit read-only EAS provider inspection with `Evidence: live-read`. This runs only `pnpm exec eas env:list --environment <env>` through the provider executor. EAS project initialization, build execution, env create/update/delete execution, and all EAS apply paths are unavailable in this slice.

Use `pnpm run provider:eas:production` to print a deterministic EAS production command plan without running provider commands. Production planning requires `EXPO_TOKEN`, prints the production EAS env/build command plan, and marks production confirmation as required for the command-plan surface. App-store submission is outside the current generated framework boundary and is not automated by this plan.

Provider command output is redacted. Env add/update/remove command plans identify variable names and whether input comes from `.agentstack/env-values.json`; Clerk env pull plans identify values as coming from Clerk Dashboard or `clerk env pull`. They do not print raw values, hashes, or secrets. Vercel env plans map missing values to `pnpm exec vercel env add`, drifted values to `pnpm exec vercel env update`, and stale values to `pnpm exec vercel env rm`. `provider plan` and `provider reconcile --plan` do not require or mutate the ledger, but print `Ledger: missing`, `Ledger: planned`, `Ledger: active`, `Ledger: blocked <status>`, or `Ledger: invalid` for supported apply targets, including the Convex preview deployment `__APP_SLUG__-preview`, Convex production deployment `prod`, and Vercel preview project `__APP_SLUG__` sections where those targets are supported.

Use `agentstack provider inventory --service <clerk|convex|vercel|eas> --env <preview|production>` to print the local provider inventory control-plane view. Local inventory is the default and is derived from the manifest, `.agentstack/provider-links.json`, and matching provider ledger rows. It prints `Evidence: local-inventory` when no matching ledger row contributes evidence and `Evidence: ledger-local-inventory` when a planned or active ledger row contributes evidence. It writes no files, does not call provider CLIs, and does not treat `.agentstack/local-cloud.json` sync links as proof of external provider existence.

Use `agentstack provider inventory --service <clerk|convex|vercel|eas> --env <preview|production> --source live` or `--live` for bounded read-only live inventory. Live inventory starts from the same local inventory row, calls only the existing provider read-only inspect primitive for the selected service/env, prints `Evidence: live-read-inventory`, `Mutation: none`, command/result counts, and redacted live status fields. When successful read-only evidence has sanitized candidate labels, standalone live inventory also prints `Candidate identity evidence`, `Candidate identity evaluator`, and a candidate-reduced `Identity proof missing` summary. With a matching planned or active ledger row, live inventory can pass the same exact proof context as live validation for the narrow Clerk preview/production application and bounded Vercel preview/production project paths; matching strict provider-owned evidence may surface `identity=matched identity-scope=exact`. Exact identity in inventory is still diagnostic only and does not prove exact drift/live coherence, readiness, link/adopt authorization, mutation permission, provisioning, or discovery. Successful Convex preview/production, Vercel preview/production, and EAS preview/production env-list reads may print sanitized partial proof labels such as `identity-scope=partial`, `facts=expected-env-names,<requested-environment>,env-list-read`, and `missing=ledger-comparable-identity,stable-provider-identity`; they still keep `identity=ambiguous` unless the narrow Clerk/Vercel exact context is proven, and do not print actual env variable names, provider IDs, URLs, tokens, raw command output, or exact project/app identity. Vercel live inventory can reduce missing-proof guidance when sanitized identity candidate labels are available: `provider-environment-scope` can come from structurally parsed env-list evidence, `stable-provider-identity`, `provider-owner-identity`, and `provider-resource-id` can come from strict provider-owned `project ls --json`, and `provider-project-link-proof` can come from a valid local `.vercel/project.json` with non-empty `projectId` and `orgId`. The local Vercel project-link candidate is local link-state evidence only unless strict provider-owned project JSON, manifest, planned or active ledger, local-link, and requested-environment env-list gates all match. Raw Vercel project IDs, org IDs, env values, and provider output are not printed. EAS candidate behavior can emit sanitized `provider-environment-scope` from structurally parsed env-list rows for the requested environment and sanitized `provider-project-link-proof` from a valid local Expo app config containing `extra.eas.projectId`. The local EAS project-link candidate is local config evidence only; EAS still does not emit exact identity proof, stable provider identity, provider IDs, project IDs, URLs, env names, env values, exact drift/live coherence, readiness, link/adopt confirmation, or mutation permission. Successful Clerk diagnostics can print `facts=diagnostics-read`; Clerk preview env/config reads can print fixed sanitized config/env facts only when strict JSON object parsing recognizes expected env-key and redirect/webhook/organization/billing categories. Loose prose, malformed JSON, top-level arrays, missing categories, and failed reads attach no env/config facts. Without exact apps-list context, these facts still keep `identity=ambiguous` in inventory and prove only bounded read access plus partial configuration category presence, not stable provider identity. Clerk identity candidate behavior is specifically limited to conservative `apps list --json` row parsing. Successful structured Convex preview/production env-list evidence may reduce missing-proof guidance with sanitized `provider-environment-scope`, but it remains read evidence and candidate guidance only; Convex production env-list evidence is not drift proof, and Convex partial drift remains preview-only diagnostic evidence, not exact drift/live coherence. It is not provider existence proof, exact provider identity proof, readiness, live-safe link/adopt confirmation, or permission to mutate. Failed reads preserve `identity-scope=none` and may print `missing=successful-live-read`. It is not broad provider discovery, provisioning, adoption, or reconciliation.

Use `agentstack provider proof --service <clerk|convex|vercel|eas> --env <preview|production> --resource-type <type> --name <name>` for a provider-specific proof contract check. Generated scripts include `provider:clerk:proof:preview`, `provider:clerk:proof:production`, `provider:convex:proof:preview`, `provider:convex:proof:production`, `provider:vercel:proof:preview`, `provider:vercel:proof:production`, `provider:eas:proof:preview`, and `provider:eas:proof:production`. Proof runs local validation first, requires a matching `planned` or `active` ledger row, rejects unsupported resource shapes before provider executor use, never reads `.agentstack/local-cloud.json`, never writes telemetry, local-cloud, provider-links, provider resources, or `docs/provider-resource-ledger.md`, and may only call the selected service's bounded read-only inventory/inspect path. Clerk preview application proof can print `Identity proof: exact` and `Exact identity evidence: available` only when strict `clerk apps list --json` parsing finds exactly one development app row whose name/slug/displayName, `resourceId` or `resource_id`, owner, environment, and stable app id satisfy the manifest and ledger comparison gates. Clerk production application proof uses the same strict parser and comparison gates for exactly one production app row, but it remains exact-identity diagnostic evidence only with no drift/live coherence proof. The Clerk preview proof path can print `Drift proof: partial` with `Drift evaluator: clerk-apps-list-preview` for exact app-list live-coherence alone, or `Drift evaluator: clerk-config-preview` when exact app-list identity is present and strict preview env/config JSON also proves only fixed sanitized env-key and redirect/webhook/organization/billing category presence. This remains partial evidence only, not exact drift or readiness. Vercel preview and production project proof can print sanitized exact identity only when strict provider-owned `project ls --json` parsing finds exactly one project row that matches the manifest, planned or active ledger proof context, local `.vercel/project.json` comparison, and a structured env-list row for the requested environment. Vercel preview proof can print `Drift evaluator: env-list-preview`; Vercel production proof can print `Drift evaluator: env-list-production` from bounded structured production env-list evidence. Both remain partial drift diagnostics only and still report live coherence blocked/refuse readiness. `.vercel/project.json` alone remains local link-state evidence only. Raw app IDs, project IDs, resource IDs, owner IDs, org IDs, row IDs, command output, URLs, env values, and secrets are not printed. Convex preview and production proof paths keep `Identity proof: ambiguous` or `unavailable`; Convex production proof can run bounded `convex env --prod list`, may emit sanitized candidate environment-scope evidence from structured expected env rows, and remains provider-read evidence only. EAS preview and production proof paths keep `Identity proof: ambiguous` or `unavailable`; EAS proof can now reduce missing-proof guidance with sanitized environment-scope candidates from structured env-list rows and sanitized project-link candidates from local Expo config, while remaining candidate/read evidence only. Proof reports print `Candidate identity evidence`, `Candidate identity evaluator`, and candidate-reduced `Identity proof missing` labels when those candidate reads help, but those fields are guidance only and never exact identity or readiness. Convex/Vercel/EAS preview env-list reads may print partial drift evidence where structured preview env-list gates pass; Vercel production is the only production env-list partial drift diagnostic in this slice. Convex partial drift is diagnostic only: Convex still has no exact provider identity, no exact drift/live coherence, no readiness proof, no live-safe link/adopt confirmation, and no mutation permission. Proof summaries also print `Live coherence: blocked` or `Live coherence: unavailable` with fixed sanitized blockers; this is a blocker diagnostic over exact identity plus drift evidence, not readiness proof, and partial drift never becomes exact live coherence. All proof paths keep `Readiness: refused` until exact drift/live coherence is proven. Exact identity requires matched comparison evidence for the corresponding stable provider, ledger, manifest, environment, and project-link gates; sanitized labels alone are not exact proof.

Use `agentstack validate --live --env <preview|production>` for aggregate non-mutating live validation across enabled providers. It runs local validation first, then prints `Evidence: live-validation`, the bounded read-only scope, per-provider inventory with `Evidence: live-read-inventory`, and per-service provider-proof summaries. It writes no `.agentstack/provider-links.json`, no `.agentstack/local-cloud.json`, no telemetry events, and no ledger changes. For Clerk preview/production application proof and Vercel preview/production project proof, live validation can pass exact proof context after the same manifest resource and planned or active ledger gates used by provider proof, so matching strict provider-owned JSON evidence can surface exact identity. Clerk preview can also surface partial app-list diagnostics, or partial config/env diagnostics when strict preview env/config JSON categories are present after exact app-list identity. Clerk production live validation is exact-identity diagnostic only and still refuses readiness. Vercel production live validation can additionally print `Drift proof: partial` with `Drift evaluator: env-list-production` from bounded production env-list evidence, but this remains partial drift only and still refuses readiness. Convex production live validation can run bounded `convex env --prod list` and may emit sanitized candidate environment-scope evidence from structured expected env rows; EAS production live validation can run the bounded production `env:list` diagnostic. Convex/EAS production reads remain candidate/read evidence only with no exact identity, drift/live coherence, or readiness proof. Non-exact provider-proof summaries can print `Candidate identity evidence`, `Candidate identity evaluator`, and candidate-reduced `Identity proof missing` labels when read-only candidate evidence helps explain remaining blockers. Provider-proof summaries include `Live coherence: blocked` or `Live coherence: unavailable` with sanitized evaluator/blocker labels. Successful reads still end with `FAIL validate --live`, `Readiness: refused`, and `Reason: proof-incomplete`; failed reads use `Reason: live-read-failed`. Provider proof readiness is refused because exact drift/live coherence is not proven for every enabled provider; live coherence, partial drift, candidate evidence, and exact identity evidence are diagnostic only and do not authorize link, adopt, mutation, or production readiness.

Use `agentstack provider link --service <clerk|convex|vercel|eas> --env <preview|production> --resource-type <type> --name <name>` to write a local link only after a matching `planned` or `active` provider ledger row exists. Local link is the default source mode (`--source local`): it prints `Evidence: ledger-local-inventory`, `Local mutation: .agentstack/provider-links.json`, `Provider mutation: none`, and `Ledger mutation: none`. It does not mutate `docs/provider-resource-ledger.md`, telemetry, local-cloud state, call provider CLIs, create provider resources, record live existence, or promote simulator state into provider truth. Missing, incomplete, invalid, blocked, cleanup-pending, cleaned, or abandoned-with-reason ledger rows block the link.

Use `agentstack provider link --service <clerk|convex|vercel|eas> --env <preview|production> --resource-type <type> --name <name> --source live` for live-safe link confirmation. This mode still requires the same valid ledger row before any live read. It then calls only the selected service's existing read-only live inventory/inspect primitive and prints `Evidence: live-read-inventory`, `Local mutation: none`, `Provider mutation: none`, and `Ledger mutation: none`. Vercel production live link can run bounded Vercel production env-list plus project-list reads; Convex production live link can run bounded `convex env --prod list`; EAS production live link can run bounded EAS production env-list before refusing ambiguous identity. Ledger-backed Clerk and Vercel live link can pass the same exact proof context used by provider proof, so matching provider-owned evidence may surface `identity=matched`. Exact identity still does not authorize the local link write: live link refuses with `FAIL provider.link.live-coherence-blocked` or `FAIL provider.link.live-coherence-unavailable` until exact live coherence exists. Ambiguous evidence refuses with `FAIL provider.link.identity-ambiguous`, failed reads refuse with `FAIL provider.link.live-read`, missing proof requirements are summarized when identity is ambiguous, and no live-link refusal writes `.agentstack/provider-links.json`. There is no `--live` shorthand for link.

Use `agentstack provider adopt --service <clerk|convex|vercel|eas> --env <preview|production> --resource-type <type> --name <name> --external-id <id-or-url> --owner <owner> --purpose <purpose> --created-by <name> --created-at <yyyy-mm-dd> --cleanup <procedure> --cleanup-trigger <trigger-or-date> --evidence <path-or-url>` to print a redacted ledger proposal for operator review. Local adopt is the default source mode (`--source local`): it prints `Evidence: local-inventory`, `Local mutation: none`, `Provider mutation: none`, and `Ledger mutation: none`. Adopt is print-only in this slice: it does not mutate the root provider ledger, `.agentstack/provider-links.json`, telemetry, local-cloud state, or provider resources. External identifiers and row-like values are redacted in output. Generated package scripts intentionally expose inventory/link only, not adopt, because adopt requires operator-specific ledger fields that cannot be represented by a generic runnable script. Add the reviewed proposal to `docs/provider-resource-ledger.md` manually before using ledger-gated link or supported apply commands.

Use `agentstack provider adopt --service <clerk|convex|vercel|eas> --env <preview|production> --source live` for live-safe adopt confirmation. This mode does not require an existing ledger row because adoption would propose a row, but it calls only the existing read-only live inventory/inspect primitive and prints `Evidence: live-read-inventory`, `Local mutation: none`, `Provider mutation: none`, and `Ledger mutation: none`. Vercel production live adopt can run bounded Vercel production env-list plus project-list reads; Convex production live adopt can run bounded `convex env --prod list`; EAS production live adopt can run bounded EAS production env-list before refusing ambiguous identity. Clerk/Vercel live adopt can use supplied `--resource-type`, `--name`, `--external-id`, and `--owner` values as exact proof context when the resource type/name also match the manifest resource. Matching provider-owned evidence may surface `identity=matched`, but exact identity still does not print a ledger proposal or authorize adoption while exact live coherence is missing: live adopt refuses with `FAIL provider.adopt.live-coherence-blocked` or `FAIL provider.adopt.live-coherence-unavailable`. Ambiguous evidence refuses with `FAIL provider.adopt.identity-ambiguous`, failed reads refuse with `FAIL provider.adopt.live-read`, ambiguous identity output includes `Identity proof requirements:`, and no live-adopt refusal writes local files, provider resources, telemetry, local-cloud state, provider links, or `docs/provider-resource-ledger.md`. There is no `--live` shorthand for adopt.

EAS env plans map missing mobile values to `pnpm exec eas env:create`, drifted values to `pnpm exec eas env:update`, and stale values to `pnpm exec eas env:delete`. Non-secret Agentstack values use EAS `plaintext` visibility; secret values use `secret` visibility. Mobile env values for EAS Build must be present in EAS server env storage, not only in local CI settings or `.env` files.

Clerk publishable keys, `CLERK_SECRET_KEY`, and `CLERK_WEBHOOK_SIGNING_SECRET` stay provider-owned. Synchronize them through the Clerk command plan and dashboard review rather than committing generated source changes that contain key material.

Supported provider apply paths require a matching `planned` or `active` ledger row before the provider executor runs. Convex preview apply requires provider `convex`, env `preview`, resource type `deployment`, and name `__APP_SLUG__-preview`. Convex production apply requires provider `convex`, env `production`, resource type `deployment`, and name `prod`. Vercel preview apply requires provider `vercel`, env `preview`, resource type `project`, and name `__APP_SLUG__`. Missing, incomplete, invalid, blocked, cleanup-pending, cleaned, or abandoned-with-reason ledger rows block mutation with `FAIL provider.ledger.*` diagnostics.

`validate --cloud --env <env>` checks linked services and provider env resource presence or drift against local-cloud state only. It prints `Evidence: local-rehearsal` and `Scope: local-cloud state only; no live provider reads`. `.agentstack/local-cloud.json` sync links are simulator state and are not proof of external provider existence. Clerk inspect, Convex preview/production inspect, Vercel preview/production inspect, and EAS preview/production inspect are read-only. Convex apply executes Convex commands, and Vercel preview apply executes only the preview deploy command. Vercel production apply, Vercel env mutation execution, EAS build/init/env mutation execution, and EAS apply remain unavailable; command output plus telemetry remain redacted. Live mutation remains explicit and ledger-gated through supported provider apply commands.
