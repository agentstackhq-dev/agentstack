Agent lab

These scenarios compress the framework's most important feedback loops into small cause-and-effect exercises. Pick a change, then read the expected failure, repair command, and agent lesson.

What breaks, what explains it, what repairs it

The lab is meant to make the framework's mechanics memorable. It shows how a coding agent should interpret validation output instead of guessing at the underlying Convex, Clerk, Vercel, EAS, or telemetry state.

The framework wants agents to ask better next questions.

A useful diagnostic is not just a red mark. It names the violated contract, points at the file or service surface, and tells the agent which command or edit moves the system back toward a coherent state.

The local provider is intentionally explicit: every failure and preview deploy artifact is a rehearsal for future Convex, Clerk, Vercel, and EAS adapters.

How to use this page

During design review, ask which contract should become mechanically checkable instead of relying on an agent remembering the convention.

  • Feature work should define the new invariant and the command that checks it.
  • Provider work should map to initialize, sync, inspect, validate, repair, and observe.
  • Production debugging should begin from correlated telemetry and narrow by journey.

Design review prompts

When adding a feature

What file, manifest entry, generated anchor, env value, or test should prove the feature is wired correctly?

When adding a provider

Which provider state can be represented locally first, and which remote operation needs a bounded wrapper?

When debugging production

Which event, trace, correlation id, user journey, or service span should an agent inspect before editing code?
