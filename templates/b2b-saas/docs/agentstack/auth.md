# Auth

Clerk owns user identity for the generated B2B SaaS template. Keep auth behavior explicit:

- Web and mobile surfaces should show sign-in first, then sign-up.
- Product routes should rely on Convex membership checks for authorization.
- Signed-in users should see continue or sign-out actions instead of duplicate auth prompts.
- Use the SaaS spine helpers in `packages/domain/src/saas-spine.ts` for role and permission checks before adding surface-specific auth logic.
- Normalize Clerk webhook ingestion through `convex/saasSpine.ts` metadata and keep provider SDK code behind wrappers.

The generated Vite web app wraps `App` with `ClerkProvider` and `ConvexProviderWithClerk` when `VITE_CLERK_PUBLISHABLE_KEY` and `VITE_CONVEX_URL` are configured. Without real values, the app renders a runtime-configuration notice instead of mounting Clerk or Convex clients. After sign-in, the web app calls `workspaceStatus.protectedStatus`, a Convex query that checks `ctx.auth.getUserIdentity()` before returning workspace status. For M1 smoke evidence, the web app exposes `data-agentstack-auth-state` (`runtime-not-configured`, `loading`, `signed-out`, `signed-in`) and `data-agentstack-protected-data-state` (`loading`, `protected-data-loaded`) markers, plus `data-agentstack-protected-workspace-id` when the protected Convex query returns. `convex/auth.config.ts` reads `CLERK_JWT_ISSUER_DOMAIN` and configures Clerk as the Convex auth provider with `applicationID: "convex"`.

Use `pnpm run provider:clerk:preview` and `pnpm run provider:clerk:production` to print the bounded Clerk command plan. Provider inspection uses read and diagnostic commands only, such as `pnpm exec clerk doctor --mode agent`, `pnpm exec clerk env pull --mode agent`, and `pnpm exec clerk config pull --mode agent`. Preview proof may use strict JSON env/config pull results for fixed sanitized category presence only after exact apps-list identity is proven; production proof is limited to exact apps-list identity diagnostics. Neither path prints raw Clerk URLs, webhook endpoints, IDs, env values, tokens, or command output. Provider plans may still render deterministic setup or review steps such as `pnpm exec clerk init -y` and the production `pnpm exec clerk deploy --mode agent` plan item separately; review those steps before running them outside Agentstack.

Use `pnpm run provider:clerk:proof:preview` only after adding a matching planned or active Clerk preview application row to the provider ledger. Use `pnpm run provider:clerk:proof:production` only after adding a matching planned or active Clerk production application row. Both paths are fail-closed proof contract checks that may run bounded read-only Clerk diagnostics, never read local-cloud state, write nothing, and can emit sanitized exact identity when strict apps-list and ledger/manifest gates pass. Preview proof can also emit partial app-list/config/env live-coherence evidence. Clerk production proof remains an exact-identity diagnostic only: it reports drift unproven/live coherence blocked and still refuses readiness until exact drift/live coherence exists.

Expected local values are documented in `.env.example`. Keep real keys out of git and prefer provider-owned Clerk secrets over copied source values. Clerk env values such as `CLERK_SECRET_KEY`, publishable keys, and `CLERK_WEBHOOK_SIGNING_SECRET` should be synchronized through the Clerk CLI plan or dashboard review rather than pasted into generated source.
