# Validation

Start with `pnpm run inspect` when you enter the generated repo. It prints the app slug, framework and guidance versions, generated anchor counts, enabled services, preview local-cloud state, provider adapter contract status, and pending provider operation IDs without mutating anything.

Provider adapters marked `contract-only` have a normalized provider boundary. Pending provider operation IDs are stable and redacted. Environment operations expose variable names only, never values or hashes. An `env.set` operation can appear before a local value is available; sync remains the actionability gate. Provider execution happens only through explicit `agentstack provider inspect/apply`; plan, sync, deploy, build, doctor, and dev commands do not execute provider CLIs.

Run `pnpm run validate` for local structural checks before completion. It checks the manifest, local env declarations, generated anchors, guidance version, theme token shape, and source secret policy. Expected output includes `Evidence: local-structure`; it does not run package quality commands, read live provider state, or prove deployment readiness.

Run `pnpm run validate:quality` before completion when code changed. It runs structural validation first, then local package quality commands: `pnpm lint`, `pnpm typecheck`, and `pnpm test`. Expected output includes `Evidence: local-quality` and the scope `local filesystem and package commands only; no local-cloud writes; no provider executor; no live provider reads`. Command failures print the command id, command, exit code, and redacted/truncated stdout/stderr tails.

Run `pnpm run validate:live:preview` or `pnpm run validate:live:production` for aggregate live provider validation when you need bounded read-only provider evidence. It runs local structural validation first; if local validation fails, it stops before provider executor use and writes no telemetry, local-cloud state, provider links, or ledger changes. On the live path it prints `Evidence: live-validation`, `Scope: bounded read-only provider inventory; no local-cloud writes; no provider mutations`, and `Mutation: none`, then reuses the same live inventory read path that prints `Evidence: live-read-inventory` and adds per-service provider-proof summaries. For Clerk preview and production, live validation passes exact proof context after the same manifest resource and planned or active ledger gates used by provider proof, so matching strict apps-list evidence can surface `Identity proof: exact`, `Exact identity evidence: available`, and `Exact identity evaluator: provider-exact-identity`. Clerk preview can additionally surface `Drift proof: partial` and either `Drift evaluator: clerk-apps-list-preview` or `Drift evaluator: clerk-config-preview` when strict preview env/config JSON also proves fixed sanitized env-key and redirect/webhook/organization/billing category presence. Clerk production live validation is exact-identity diagnostic only and still refuses readiness. For Vercel preview and production, strict provider-owned `project ls --json` evidence can surface sanitized exact identity only when one project row matches the manifest, ledger, local project-link comparison gates, and requested-environment env-list scope. Vercel production live validation is exact-identity diagnostic only and still refuses readiness. Convex, Vercel, and EAS preview env-list reads can surface sanitized partial drift evidence where structured env-list gates pass, but this remains diagnostic and never exact live coherence. Convex production env-list reads can surface only `provider-env-read`, and EAS production env-list reads can surface only candidate production-scope evidence; they do not prove exact identity, drift/live coherence, or readiness. This command is non-mutating and currently refuses readiness with `FAIL validate --live`, `Readiness: refused`, and `Reason: proof-incomplete` while exact drift/live coherence is not proven for every enabled provider. Partial or exact identity evidence is diagnostic only and does not authorize link, adopt, mutation, or production readiness. Failed live reads refuse with `Reason: live-read-failed`.

Run the per-service `provider:*:proof:preview` scripts plus `provider:clerk:proof:production`, `provider:convex:proof:production`, `provider:vercel:proof:production`, or `provider:eas:proof:production` only when a matching planned or active provider ledger row exists and you need a fail-closed provider proof contract check. Proof runs local validation first, rejects unsupported resource shapes before executor use, may run bounded read-only inspect/inventory, prints `Evidence: live-proof-check`, never reads local-cloud state, and writes nothing. Clerk preview application proof can emit sanitized exact identity and partial app-list live-coherence drift evidence only after strict apps-list parser and ledger/manifest comparison gates pass; if strict preview env/config JSON also proves fixed env-key and redirect/webhook/organization/billing categories, the partial evaluator can become `clerk-config-preview`. Clerk production application proof can emit sanitized exact identity from a strict production apps-list row and matched ledger/manifest comparison evidence, but it remains exact-identity diagnostic only with drift unproven and live coherence blocked. Convex production proof can run bounded `convex env --prod list`, but remains read evidence only with no exact identity, drift/live coherence, or readiness. Vercel preview and production project proof can emit sanitized exact identity after strict provider-owned project JSON plus manifest, ledger, local-link, and matching environment-scope gates pass. Vercel production project proof can also emit `Drift evaluator: env-list-production` from bounded structured production env-list evidence, but this is partial drift only; it is not link/adopt authorization and does not prove Vercel exact drift/live coherence or readiness. EAS production proof can run bounded production `env:list`, but remains candidate/read evidence only with no exact identity, drift/live coherence, or readiness. Convex, Vercel, and EAS preview env-list reads can emit sanitized partial drift evidence where structured env-list gates pass; Convex partial drift is diagnostic only and still leaves Convex without exact provider identity, exact drift/live coherence, readiness proof, live-safe link/adopt confirmation, or mutation permission. Proof summaries also print `Live coherence: blocked` or `Live coherence: unavailable` with fixed sanitized blockers; this is a blocker diagnostic, not readiness proof. Convex/EAS exact proof remains missing or candidate-only as applicable, and every proof path refuses readiness until exact drift/live coherence is proven.

Run `pnpm run doctor` before provider, environment, build, sync, or deploy work. It runs local validation plus preview local-cloud checks and prints repair commands before you open a provider workflow.

Run `pnpm run dev` as a local preflight when you need the next commands for product work. It should point you toward the generated workspace-status web, mobile, and Convex surfaces plus validation, env, and sync commands. It must not mutate provider state.

Run `pnpm run theme:validate` when editing `packages/theme/tokens.json` or `packages/theme/src/index.ts`. The normal local validation gate also includes theme diagnostics and blocks completion when required token paths are missing or invalid.

Required custom env declarations are satisfied from `.agentstack/env-values.json` using the environment -> surface -> variable shape documented in `docs/agentstack/environments.md`. `agentstack env set` writes that local validation state only; it does not create or update provider resources.

Local validation also scans source, docs, and `.env` files for raw secret-like values. Use `agentstack sync --env <env>` for local provider env resource rehearsal after values exist. Sync refuses missing or invalid declared values before planning or applying provider env resources. Applied sync records provider env resources in `.agentstack/local-cloud.json` with redacted/hash-only metadata such as `valueHash`, never raw env values. Clerk, Convex, Vercel, and EAS command-plan adapters print bounded provider CLI command shapes with `Evidence: provider-command-plan`. `pnpm run provider:preview:plan` and `pnpm run provider:production:plan` run the same local validation gate first and then print all enabled service command plans for the selected environment with `Provider execution: none`, `Mutation: none`, and `Readiness: not-claimed`; failed local validation stops before service plan sections. `pnpm run provider:preview:reconcile` and `pnpm run provider:production:reconcile` print `Evidence: provider-reconciliation-plan`, `Provider execution: none`, `Mutation: none`, `Readiness: not-claimed`, `Live state: not-read`, `Local-cloud state: not-read`, and `Operations: not-evaluated`; they use local validation and sanitized ledger summaries only, do not call provider executors or provider CLIs, do not read live provider state, local-cloud rehearsal state, or provider links, write no `.agentstack/events.jsonl`, `.agentstack/local-cloud.json`, `.agentstack/provider-links.json`, provider resources, or `docs/provider-resource-ledger.md`, and do not claim exact identity, drift proof, provisioning, adoption/link confirmation, live coherence, or readiness. `provider plan` and `provider reconcile --plan` can print sanitized ledger summaries for supported apply targets without requiring or mutating the ledger. Clerk inspect, Convex preview/production inspect, Vercel preview/production inspect, and EAS preview/production inspect are explicit provider diagnostics with `Evidence: live-read`. Supported provider apply paths are ledger-gated before execution, print `Evidence: live-mutation`, and are limited to Convex apply plus Vercel preview deploy.

Run `pnpm run env:inspect` when provider state, environment sync, or deployment readiness needs deeper detail after `pnpm run doctor`. Use `pnpm run preview:plan` to plan local-cloud repair, `pnpm run preview:apply` to apply it, `pnpm run preview:validate` to compare expected preview state with configured services and provider env resources, and `pnpm run preview:deploy` to rehearse deploy planning locally. `validate --cloud` reports missing, stale, or drifted provider env resources through the same local-cloud contract that command-plan adapters read. It prints `Evidence: local-rehearsal` and `Scope: local-cloud state only; no live provider reads`; it does not call the provider executor and does not prove that external provider resources exist.

`provider-reconciliation-plan` is a separate evidence category from structural validation, local quality validation, local-cloud rehearsal, and live validation. It is a typed aggregate plan artifact, not provider read evidence and not readiness proof. Live validation is provider read evidence plus an explicit readiness refusal, not production readiness proof.

Use `pnpm run mobile:build:development`, `pnpm run mobile:build:preview`, and `pnpm run mobile:build:production` to validate generated mobile build profiles. The preview apply step writes `.agentstack/builds/mobile-preview.json` only; it does not call Expo or EAS. EAS runtime execution is limited to explicit preview/production `env:list` inspection; project initialization, builds, env mutation execution, and apply are unavailable.

Generated scripts delegate to an installed `agentstack` CLI. For source-prototype smoke runs, set `AGENTSTACK_CLI_BIN` to the framework CLI entrypoint and `AGENTSTACK_TSX_BIN` to the local `tsx` runner before running package scripts.
